## 2. 暂定标题
**面向日频模拟交易的多 Agent 股票决策支持系统设计与实现**

## 3. 问题描述或使用场景描述
股票分析与交易决策通常需要综合多种异构信息，包括历史价格走势、技术指标、财经新闻、公司基本面以及风险约束。对于进行投资研究或模拟交易的用户而言，这些信息来源分散、形式不一致，人工整理成本较高，而且不同来源的信息还可能相互冲突，导致决策过程复杂且缺乏透明性。

本项目拟构建一个基于大语言模型的股票决策支持系统，面向**日频模拟交易**场景，为用户提供结构化的买入、卖出、持有或调仓建议。系统不接入真实券商账户，也不进行实盘自动执行，而是在历史回测或模拟环境中运行，重点服务于投资研究和交易决策辅助。

为保证项目范围清晰且便于评估，我们将系统任务限定为以下三类代表性任务：

1. **单股票分析任务**：给定某只股票近期价格、新闻和基本面信息，系统输出交易建议、支持理由和风险提示。  
2. **候选股票筛选任务**：给定一个小型股票池，系统识别相对值得关注或需要规避的标的。  
3. **小型股票池调仓任务**：在给定持仓和风险约束条件下，系统生成调仓建议并说明依据。  

本项目的目标不是构建自动执行交易的“炒股机器人”，而是实现一个面向模拟交易与投资研究的、**可解释、可验证、可交互的决策支持原型系统**。

## 4. 选择的路线
**Applied Track（应用路线）**

我们选择应用路线，因为本项目的主要贡献在于构建一个可运行的软件系统，围绕股票分析这一具体场景，设计其多模块协作流程、信息组织方式、风险提示机制和用户可解释输出。项目重点是系统设计与应用验证，而不是提出新的金融理论或围绕单一研究假设开展严格实验。

## 5. 核心技术思路或系统设计
本项目计划设计一个**多 Agent 协作的股票决策支持系统**。系统不采用单一 Agent 直接输出结论，而是将金融分析流程拆分为多个功能角色，以提高系统的结构化程度、可解释性和稳定性。

系统初步包含以下模块：

### （1）市场信息 Agent
负责读取股票历史价格、成交量及技术指标，识别趋势、波动和短期异常变化，输出市场侧分析结果。

### （2）新闻与情绪 Agent
负责处理与目标股票或市场相关的财经新闻标题、摘要或事件信息，提取潜在利好、利空和事件驱动信号，并对新闻情绪进行归纳。

### （3）基本面分析 Agent
负责读取公司公开财务指标，例如营收、净利润、EPS 及部分资产负债表项目，对公司中长期经营状态进行总结。

### （4）风险管理 Agent
负责检查系统建议是否符合预设约束，包括仓位上限、单只股票集中度、回撤风险和交易频率限制等；在必要时对高风险建议进行修正或提示。

### （5）决策 Agent
负责综合前述模块输出，给出最终建议，包括：
- 买入 / 卖出 / 持有 / 调仓
- 建议依据
- 风险提示
- 关键支持信息摘要

除分析模块外，系统还将设计一个基础的**结果展示层**，使用户能够看到各模块的中间分析结果，而不仅是最终结论。这样既有助于用户理解系统形成建议的过程，也更符合投资研究中的实际使用习惯。

## 6. 暂定评估方案
本项目的评估重点不在于证明某个理论假设，而在于验证系统是否能在目标使用场景下**稳定、合理、可解释地工作**。我们将从以下四个方面进行评估。

### （1）代表性任务测试
围绕前述三类任务进行测试：
- 单股票分析
- 候选股票筛选
- 小型股票池调仓建议

每类任务都会选取若干代表性样例，以检查系统是否能完成任务并输出结构化结果。

### （2）预定义成功标准
为了避免评估过于模糊，我们将预先设定以下成功标准：

- 在**单股票分析任务**中，系统应能输出明确建议，并给出支持理由和至少一条风险提示；
- 在**候选股票筛选任务**中，系统应能对多个标的进行区分，而不是对所有股票给出相似结论；
- 在**调仓任务**中，系统不应违反预设仓位或集中度约束；
- 在**信息冲突或不完整场景**下，系统应能显式表达不确定性，而不是强行给出过度自信的激进建议；
- 系统输出应具备基本可解释性，即用户能够从中间模块结果中理解最终建议的主要依据。

### （3）与简单 baseline 对比
我们计划与以下简单版本进行比较：
- 单一 Agent 直接读取全部输入并输出建议；
- 不包含风险管理模块的简化版本；
- 仅基于简单技术指标规则的基础策略。

比较重点不是单纯追求更高收益，而是考察：
- 输出是否更完整；
- 风险提示是否更充分；
- 面对复杂输入时是否更稳定；
- 决策理由是否更清晰。

### （4）结构化流程测试与失败案例分析
我们还将检查系统在端到端流程中的表现，例如：
- 是否能完成价格读取、新闻分析、基本面总结和最终整合；
- 是否会忽略模块间冲突；
- 是否能在风险较高时主动发出警告；
- 是否能在缺乏足够依据时给出保守结论。

此外，我们将对失败案例进行分析，例如新闻噪声、基本面缺失、市场剧烈波动或模块结论相互矛盾等情况，以说明系统的局限性。

如果时间允许，我们还将补充基础回测指标，如累计收益率、年化收益率、最大回撤和夏普比率，作为辅助参考。

## 7. 计划使用的数据来源、基准、用户群体或任务集
为保证项目可行性，我们将采用**小规模股票池 + 日频历史数据 + 多源金融信息融合**的任务设置，而不涉及高频交易或实时市场接入。

### （1）股票价格数据
我们计划使用 Kaggle 的 **S&P 500 Stocks** 数据集作为主要价格数据来源。该数据集提供标普 500 成分股的历史股票与公司数据，适合提取日频 OHLCV 信息。项目中我们不会使用全部股票，而是从中选取 **10–20 只美国大盘股** 作为小型股票池，用于控制任务范围和实验复杂度。

### （2）新闻数据
考虑到仅包含 headline 的数据难以支撑新闻理解、事件归因和可解释理由生成，我们不再使用只含标题的新闻汇总数据。我们计划将 **Alpha Vantage 的 News & Sentiment API** 作为主要新闻侧输入来源，按股票代码和时间范围获取历史新闻与情绪信息；同时补充使用 **Alpha Vantage 的 Earnings Call Transcript** 数据，获取公司财报电话会文本，作为较长文本输入，以增强系统对公司事件和管理层表述的理解。如果后续需要加强个股级新闻覆盖，我们将以 **Finnhub Company News API** 作为补充来源，按目标股票与日期区间抓取公司新闻。

### （3）基本面数据
我们计划使用美国证券交易委员会 SEC 的 **EDGAR Company Facts API / companyfacts.zip** 作为基本面数据来源。该接口和批量归档数据可以提供公司层面的 XBRL 财务事实数据，并支持按公司聚合财务概念。我们将从中抽取营收、净利润、EPS 等关键财务指标，用于基本面分析模块。相比第三方非官方整理数据，SEC 官方来源更稳定，也更便于说明数据可追溯性。

### （4）任务集构造方式
在任务设计上，我们将围绕上述三类任务构造样例：
- **单股票分析**：选取某股票在特定交易日之前一段窗口内的价格、新闻和最新可用基本面数据；
- **候选股票筛选**：从小型股票池中选取若干股票，要求系统给出排序或优先关注对象；
- **调仓任务**：在给定初始持仓和风险约束下，要求系统生成新的调仓建议。

目标用户主要包括：
- 进行投资研究的学生；
- 需要模拟交易辅助分析的初学者；
- 希望理解交易建议依据与风险提示的用户。

本项目不面向真实实盘自动化执行，而是服务于**投资研究和模拟交易支持**这一更适合课程范围的场景。

## 11. 一个关键风险、可行性问题或范围控制挑战
本项目最大的挑战是**范围控制与评估边界设定**。

股票交易系统很容易扩展到实时数据接入、券商 API、实盘执行、多市场、多资产和高频策略等复杂方向。如果范围控制不当，项目将很难在课程周期内完成，也难以保证评估的清晰性和可复现性。

因此，我们将主动限制项目范围：
- 只做**历史回测 / 模拟交易**，不接入实盘；
- 只做**日频决策**，不处理高频数据；
- 只使用**10–20 只股票**的小规模股票池；
- 重点关注系统设计、解释输出和风险提示，而非追求复杂策略或极端收益。

此外，金融任务本身存在较高不确定性，系统给出“合理建议”并不等于能够稳定获得高收益。因此，我们会避免将项目目标表述为“预测市场”或“保证盈利”，而是将其定位为**面向模拟交易与投资研究的决策支持工具**。
